## 2. Назначение и цели создания системы

### 2.1. Проблема, решаемая проектом
Существующая система не позволяет эффективно управлять загрузкой пунктов выдачи заказов (ПВЗ), что приводит к перегрузкам в пиковые часы, возникновению очередей и простоям в периоды низкой активности.

### 2.2. Целевая аудитория
1. Операторы ПВЗ (линейный персонал).
2. Супервайзеры регионов (менеджмент среднего звена).
3. Операционные аналитики (стратегическое планирование).
4. Клиенты маркетплейса (косвенная аудитория, получающая выгоду от отсутствия очередей).

### 2.3. Назначение системы
Система ПВЗ Monitor предназначена для мониторинга, анализа и оптимизации работы пунктов выдачи заказов за счёт предиктивного анализа и оперативного информирования о перегрузках.

### 2.4. Цель создания системы
Целью создания системы является разработка MVP для балансировки нагрузки на точки выдачи, минимизации очередей в пиковые часы и повышения эффективности работы персонала.

### 2.5. Задачи проекта
Для достижения поставленной цели необходимо решить следующие задачи:
1. Разработать модуль сбора и хранения данных об операциях ПВЗ.
2. Реализовать алгоритм расчёта текущей загрузки относительно паспортной мощности.
3. Создать интерфейс для визуализации данных (таблицы, графики, тепловые карты).
4. Обеспечить возможность экспорта аналитических отчётов.

### 2.6. Границы MVP
В состав MVP входят следующие возможности:
1. Мониторинг загрузки по часам и дням.
2. Выявление перегрузок относительно паспортной пропускной способности.
3. Визуализация данных через графики и тепловую карту.
4. Экспорт данных в формат CSV.

### 2.7. Что не входит в MVP
В текущую версию проекта не входят:
1. Интеграция с реальными геосервисами.
2. Прямое управление терминалами выдачи.
3. Мобильное приложение для клиентов.

## 3. Характеристика объекта автоматизации

### 3.1. Краткая характеристика предметной области
Предметная область охватывает логистические процессы «последней мили» в e-commerce, связанные с приемом, хранением и выдачей заказов в специализированных пунктах (ПВЗ).

### 3.2. Текущий способ решения задачи
В настоящее время мониторинг осуществляется постфактум на основе общих отчетов о продажах, без оперативного контроля загрузки в реальном времени и учета пропускной способности конкретных точек.

### 3.3. Ограничения предметной области
1. Технические: использование SQLite для MVP.
2. Организационные: необходимость соблюдения графика работы ПВЗ.
3. Временные: жесткие рамки учебного семестра.

## 5. Описание основного бизнес-процесса

### 5.1. Наименование основного процесса
Регистрация и классификация операции ПВЗ

### 5.2. Краткое описание процесса
Процесс начинается с запроса на регистрацию операции (прием/выдача), включает валидацию по расписанию, расчет текущей загрузки часа и завершается записью в БД с присвоением статуса состояния (Норма/Перегрузка).

### 5.3. Участники процесса
В выполнении процесса участвуют следующие роли:
1. Оператор
2. Система (автоматизированный расчет)

### 5.4. Шаги процесса

| № | Роль | Действие | Входные данные | Результат |
|---|------|----------|----------------|-----------|
| 1 | Оператор | Инициирует операцию | Тип операции, ID ПВЗ | Запрос в систему |
| 2 | Система | Валидирует контекст | Время, расписание | Статус валидности |
| 3 | Система | Рассчитывает загрузку | Кол-во операций в час | % загрузки |
| 4 | Система | Классифицирует состояние | % загрузки, пороги | Уровень (Норма/Крит) |
| 5 | Система | Регистрирует данные | Все параметры операции | Запись в БД |

### 6.3. Функциональные требования по ролям

#### 6.3.1. Требования к роли Оператор
1. Система должна позволять регистрировать новую операцию.
2. Система должна отображать текущий статус загрузки своего ПВЗ.
3. Система должна сохранять результат операции в БД.

#### 6.3.2. Требования к роли Супервайзер
1. Система должна позволять просматривать список ПВЗ региона.
2. Система должна отображать KPI по перегрузкам для региона.
3. Система должна изменять фильтры отображения данных.

#### 6.3.3. Требования к роли Аналитик
1. Система должна позволять видеть данные по всей сети ПВЗ.
2. Система должна предоставлять функционал экспорта данных и тепловую карту.

### 6.4. Ограничения реализации функциональности
1. Расчет загрузки производится дискретно по часовым интервалам.
2. Валидация операций выполняется строго на стороне сервера.
3. Данные в интерфейсе обновляются при перезагрузке страницы или по триггеру.

## 7. Требования к данным

### 7.1. Основные сущности системы
В системе должны использоваться следующие сущности:
1. ПВЗ (pvz)
2. Расписание (schedule)
3. Операция (operations)
4. Лог ошибок (error_log)
5. Пользователь (users)

### 7.2. Описание сущностей

| Сущность | Назначение | Основные атрибуты | Первичный ключ |
|----------|------------|-------------------|----------------|
| pvz | Справочник пунктов выдачи | address, capacity_per_hour, region | pvz_id |
| schedule | График работы ПВЗ | weekday, open_time, close_time | id |
| operations | Журнал транзакций | ts, type, pvz_id | op_id |
| error_log | Журнал ошибок | reason, op_type, logged_at | id |
| users | Учетные записи | login, password, role | id |

### 7.3. Связи между сущностями

| № | Сущность 1 | Сущность 2 | Тип связи | Описание |
|---|------------|------------|-----------|----------|
| 1 | pvz | schedule | 1:M | Один ПВЗ имеет расписание на каждый день недели. |
| 2 | pvz | operations | 1:M | Один ПВЗ содержит множество операций. |
| 3 | pvz | error_log | 1:M | Ошибки логируются в привязке к ПВЗ. |

### 7.4. Ограничения данных
1. Уникальность pv_id в справочнике ПВЗ.
2. Обязательность поля ts (timestamp) для каждой операции.
3. Диапазон загрузки от 0% до бесконечности (норма < 100%).
4. Соответствие типа операции списку (in, out, return).
5. Статус ПВЗ определяется динамически на основе расписания.

## 8. Нефункциональные требования

### 8.1. Требования к интерфейсу
1. Интерфейс системы должен быть простым и единообразным (Vanilla JS + CSS).
2. Основные действия пользователя должны выполняться не более чем за 3 шага.
3. На экране должны отображаться графики Chart.js и иконки Lucide.

### 8.2. Требования к производительности
1. Время отклика основных страниц не должно превышать 500 мс.
2. Система должна корректно работать при нагрузке до 100 одновременных запросов.

### 8.3. Требования к надёжности
1. Данные должны сохраняться в файле pvz.db после перезапуска приложения.
2. Система не должна допускать потерю данных при некорректном вводе (валидация).

### 8.4. Требования к безопасности
1. Пользователь должен иметь доступ только к тем функциям, которые соответствуют его роли.
2. Система должна обеспечивать проверку корректности ввода данных (5 критериев валидации).

### 8.5. Требования к доступности и удобству использования
1. Интерфейс должен быть понятен пользователю без длительного обучения.
2. Сообщения об ошибках должны быть сформулированы понятно (reason в логе).

## 10. Критерии приёмки MVP

### 10.1. Общие критерии приёмки
MVP считается реализованным, если:
1. Реализована авторизация для всех 3 ролей.
2. Система корректно рассчитывает загрузку на основе данных БД.
3. Отображается тепловая карта загрузки.
4. Работает экспорт в CSV.
5. Пройдены автоматизированные тесты (pytest).

### 10.2. Проверка функциональных требований

| Код требования | Способ проверки | Ожидаемый результат |
